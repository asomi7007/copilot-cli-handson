# 07 — 스킬(Skills) 작성

지침이 항상 적용되는 규칙이라면, **스킬(Skills)** 은 여러 단계로 된 워크플로우를 패키징한 것입니다. 사용자가 명령을 외우지 않아도, 요청이 일치하면 에이전트가 알아서 호출합니다. 이 모듈에서는 "생성한 게임을 갤러리에 업로드하는" 스킬을 직접 만듭니다.

## 스킬이란

스킬은 `SKILL.md` 파일과, 선택적으로 보조 스크립트·참고자료·템플릿로 이루어진 폴더입니다. 마크다운 본문은 자연어 지시(언제 적용하고, 어떤 단계를 따르고, 성공이 무엇인지)이고, 상단 설정 헤더의 **이름** 과 특히 **설명** 이 Copilot이 스킬의 적용 시점을 판단하는 기준입니다.

스킬이 위치하는 곳에 따라 사용 범위가 정해집니다.

- **프로젝트 스킬** — 저장소의 `.github/skills/<이름>/` 에 위치, 저장소에서 작업하는 모두가 사용
- **개인 스킬** — 홈 디렉터리의 `~/.copilot/skills/<이름>/` 에 위치, 모든 프로젝트에서 본인이 사용
- **플러그인 스킬** — 플러그인에 번들로 포함(04 모듈의 share-screenshot이 이 방식)

스킬이 프롬프트보다 강력한 이유는 **지시와 함께 코드를 포함**할 수 있다는 점입니다. `SKILL.md` 가 옆에 있는 스크립트(bash, PowerShell, Python 등)를 실행하도록 지시해, API 호출이나 파일 처리 같은 정확한 작업을 위임할 수 있습니다. 모델은 "언제 호출할지" 판단하고, 스크립트는 결정적인 "어떻게"를 담당합니다.

스킬은 주로 **의도로 호출**됩니다. 사용자의 요청이 설명과 일치하면 Copilot이 자동으로 선택합니다. 동시에 `/스킬이름` 형태의 명령으로도 노출되므로, 명시적으로 호출할 수도 있습니다.

좋은 스킬과 나쁜 스킬의 차이는 다음과 같습니다.

- 나쁜 스킬: 설명이 모호해(예: "업로드한다") Copilot이 선택하지 못하거나, 스크립트로 처리할 로직을 마크다운에 욱여넣습니다.
- 좋은 스킬: 의도가 분명한 설명, 짧고 명확한 단계, 그리고 결정적 작업을 작은 스크립트에 위임합니다. `SKILL.md` 는 간결하게 유지하고, 긴 참고자료는 `references/`, 템플릿은 `assets/`, 실행 로직은 `scripts/` 에 둡니다.

## 작성 절차

`SKILL.md` 와 스크립트를 직접 쓰는 대신 `/create-skill` 명령으로 생성합니다.

> 업로드 API 규격을 미리 알려줄 필요가 없도록, 공용 갤러리는 업로드 API의 OpenAPI 명세를 공개합니다. 에이전트가 이 명세를 읽어 올바른 엔드포인트·메서드·요청 형식을 파악합니다.

작업 폴더에서 새 대화를 시작한 뒤 다음을 입력합니다.

```
/create-skill 생성한 우주 침략자 게임을 갤러리에 올리는 스킬을 만들어줘. 사용자는 표시할 이름과 게임이 담긴 HTML 파일 이름을 제공하고(최대 200KB), 스킬은 Windows(PowerShell)와 Mac(bash) 양쪽에서 동작하는 업로드 스크립트를 포함해야 해. 업로드 API 규격은 https://rg-hub-web-0614.azurewebsites.net/api/openapi.json 의 OpenAPI 명세를 따르고, SKILL.md는 HTTP 호출을 직접 하지 말고 스크립트를 실행하도록 지시해줘.
```

> 위 주소는 예시입니다. 진행자가 다른 갤러리 주소를 안내했다면 그 주소로 바꿉니다.

Copilot이 몇 가지를 되묻거나 도구 실행 권한을 요청할 수 있습니다. 답하거나 허용한 뒤, 제안된 스킬을 검토하고 저장(Keep)합니다. 생성된 스킬은 다음을 갖춰야 합니다.

- 작업 공간 아래에 위치(팀 전체가 사용)
- `SKILL.md` 에 이름과 의도가 분명한 설명(예: "게임 올려줘", "갤러리에 공유" 같은 요청을 인식)
- 사용자가 제공할 입력 명시(표시 이름, HTML 파일 이름)
- 200KB 크기 제한 적용
- 업로드 API 엔드포인트 참조
- 실제 업로드를 수행하는 보조 스크립트, 그리고 SKILL.md가 그 스크립트를 호출하도록 지시

> 스킬의 효과를 높이는 두 가지: (1) **정확한 설명** — Copilot이 요청과 대조해 호출 여부를 판단하는 기준입니다. 모호한 설명("무언가 업로드")은 무시되고, 의도가 분명한 설명("생성한 게임을 갤러리에 제출")은 안정적으로 선택됩니다. (2) **결정적 작업을 스크립트로** — 마크다운은 무엇을·언제, 스크립트는 어떻게를 담당하게 해, 파일 크기·엔드포인트·오류 처리를 매번 즉석에서 만들지 않도록 합니다.

## 사용

명령 없이 자연어로 호출합니다.

1. 새 대화 시작:
   ```
   /clear
   ```
2. 요청:
   ```
   내 게임을 갤러리에 올려줘. 파일은 #index.html 이고 이름은 "홍길동의 우주 침략자" 야.
   ```
   `#` 를 입력하면 파일 이름이 자동완성되며, 해당 파일을 맥락으로 첨부합니다. "홍길동"은 본인 이름으로 바꾸고, 파일명이 다르면 그에 맞게 수정합니다.
3. Copilot이 의도를 인식해 새 스킬을 선택하고, 파일 존재 확인 → 200KB 이하 확인 → 보조 스크립트 호출 순으로 진행합니다.
4. 업로드가 성공하면 진행자가 안내한 갤러리(예: `https://.../gallery`)에서 04 모듈의 스크린샷과 함께 표시됩니다.

> Copilot이 스킬을 자동으로 선택하지 못하는 가장 흔한 원인은, 설명이 실제 사용자의 표현과 일치하지 않는 경우입니다. `SKILL.md` 의 설명을 실제로 사용한 표현이 포함되도록 넓혀 다시 시도합니다.

> 명시적으로 호출하려면 `/` 를 입력해 명령 목록에서 새 스킬을 선택합니다.

## 정리

`/create-skill` 로 프로젝트 범위 스킬(`SKILL.md` + 업로드 스크립트)을 만들고, 자연어 요청으로 게임을 갤러리에 제출했습니다. OpenAPI 명세를 가리켜 실제 API를 정확히 대상으로 삼았고, 200KB 제한과 필수 입력을 스크립트에 적용해 실행마다 동작이 일관되게 유지됩니다.

이로써 두 가지 핵심 구성 요소를 직접 다뤘습니다. 항상 적용되는 규칙인 **지침**, 그리고 의도로 호출되는 여러 단계 워크플로우인 **스킬** 입니다.
